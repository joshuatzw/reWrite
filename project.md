### 2026-07-11 (paste sometimes types a literal "v" instead of pasting — second real-device bug report, fixed)

Second real-device report: intermittently, instead of the rewritten text landing in the target app, a literal "v" character appeared. Root cause in `clipboard.rs`'s `paste_and_restore`/`paste_html_and_restore`: the synthetic Cmd+V (Ctrl+V on Windows) sequence pressed the modifier and clicked the paste key back-to-back with no settle delay between them — `enigo.key(modifier, Direction::Press)?; enigo.key(paste_key, Direction::Click)?;`. If the OS/target app hadn't yet registered the modifier as held by the time the paste-key press arrived, the target app sees an unmodified keystroke and types the literal character instead of running the paste shortcut — silently discarding the correctly-rewritten text already sitting in the clipboard right next to it. This exact race was already identified and fixed for the COPY side of this same file (`capture_selection`'s 40ms settle delay before the `c`/`copy_key` press, with a comment explaining precisely this), but the analogous fix was never applied to the paste side — a pre-existing asymmetry, not something introduced by this session's macOS work (present in `HEAD`, not just the uncommitted diff). Fixed by mirroring the copy path's timing exactly in both paste functions: press modifier → sleep 40ms → press paste key → sleep 20ms → release paste key → release modifier. Not macOS-specific — this path is shared by Windows and macOS, so the fix benefits both, though the report that surfaced it came from macOS testing. `cargo check` passes; not yet re-verified against a real paste in a real app.

### 2026-07-11 (macOS selection bubble — first real-device bug report, fixed)

First real-device feedback on the macOS build: the bubble appeared (correctly positioned) but was completely unclickable — "nothing happens" on click, every time. Root cause confirmed against Tauri 2.11.3's own source (`WebviewWindowBuilder::accept_first_mouse`, wrapping macOS's `NSWindow`/`WKWebView` "accepts first mouse" behavior): the `bubble` and `bubble_menu` windows are deliberately built with `.focused(false)` (so showing them doesn't steal keyboard focus from the source app) and neither `show_bubble` nor `show_bubble_menu` ever calls `.set_focus()` on them (unlike `show_overlay`/`show_settings`, which do). On macOS, a click on a window that isn't key is consumed purely for window activation and is never delivered to the webview's JS unless the window opts in via `acceptsFirstMouse`. Tauri exposes this as `WebviewWindowBuilder::accept_first_mouse(bool)`. Fixed by adding `.accept_first_mouse(true)` to both window builders in `lib.rs` (the bubble's pre-warm block and the bubble menu's pre-warm block) — this makes clicks land on the very first click even while the window stays non-key/non-focus-stealing, which is exactly the combination this feature needs. Windows is unaffected (its click-to-activate model doesn't have this restriction; the flag is a no-op there per Tauri's docs).

This is the first bug this session's process (implement → critical review → fix) did NOT catch, because it's a real-device-only symptom — nothing in static code review flags "this window will fail to receive its first click" the way memory-ownership or coordinate-unit bugs can be spotted by reading FFI signatures. Recorded here as a concrete argument for why the Known Gaps entries below (all the "needs real-device verification" items) are not just process boilerplate.

**Not yet re-verified on a real device** — the fix is applied and compiles clean (`cargo check`), but hasn't been re-tested against the actual click-then-menu-then-rewrite flow yet.

### 2026-07-11 (macOS selection bubble — critical-review fixes, three bugs)

An independent critical review of the Phase 4 gap-closing pass (previous entry below) found three real bugs — one HIGH severity, two MEDIUM — in `selection_watcher.rs`'s mac backend and `lib.rs`'s bubble/menu positioning. All three fixed; no other files changed.

- **Bug 1 (HIGH, fixed): the mac tap's start/stop lifecycle used bare atomics (`RUNNING`/`STOP_REQUESTED`/`RUN_LOOP`/`TAP_REF`) instead of the `Mutex<TapState>` generation-tracking pattern `esc_hook.rs`'s mac backend already proved out (and fixed twice this session — see that file's two "critical-review fixes" entries below).** Two concrete failure modes were possible: (1) a lost-stop revival, where `stop()`'s `STOP_REQUESTED.store(false)`-outside-any-lock in a fresh `start()` could land in the narrow window between an old dying tap thread waking up and re-checking its loop condition, letting the old thread observe `RUNNING=true, STOP_REQUESTED=false` and keep running alongside a second, newly-spawned tap — two live system-wide event taps, duplicate probing, a leaked thread; (2) a stale-generation stomp, where an old thread's unconditional teardown (`TAP_REF.store(null)`/`RUN_LOOP.store(null)`/`RUNNING.store(false)`) could clobber a newer generation's already-published state, and the new thread's own `while RUNNING` check would then see `false` and exit immediately — the bubble going silently, invisibly dead for the rest of the session. Concretely triggerable by rapidly toggling the "Selection bubble" Settings switch, or a fast Accessibility revoke→grant while `bubble_enabled`. Fixed by porting `esc_hook.rs`'s `mod mac` `TapState` enum (`Idle`/`Starting`/`Running(usize)`) and `Mutex<TapState>` verbatim: all state transitions are now serialized under one lock, `STOP_REQUESTED` is reset to `false` in the same critical section as publishing `Running` (not from `start()` outside any lock), and `run_tap_thread`'s teardown gates every state-clearing store on a generation check (`matches!(*state, TapState::Running(ptr) if ptr == rl as usize)`) exactly like `esc_hook.rs` does. The worker thread/channel setup in `start()` was also moved to only happen after winning the `Idle` → `Starting` transition under the same lock, so a racing concurrent `start()` can no longer spawn a second, orphaned worker either (a related failure mode the bare-atomics version was equally exposed to, not explicitly named by the reviewer but the same root cause).
- **Bug 2 (MEDIUM, fixed): Cmd+A (select-all) detection read the wrong `CGEventField`, so it never worked.** The code read modifier flags via `CGEventGetIntegerValueField(event, 12)`, but per Apple's `CGEventTypes.h`, field 12 is `kCGScrollWheelEventDeltaAxis2` — there is no `CGEventField` enumerator for modifier flags at all. Reading field 12 on a keyDown always returns 0, so `flags & FLAG_MASK_COMMAND` was always false and Cmd+A was silently never detected — a from-scratch select-all with no preceding drag never triggered a recheck, so the bubble never appeared for it (drag-selection still worked fine, masking the gap). This was a real parity gap vs. the Windows backend, whose Ctrl+A path does work. Fixed by declaring the correct, separate accessor `CGEventGetFlags(event: CGEventRef) -> u64` (not read via `CGEventGetIntegerValueField`) and calling that instead; the now-unused `FIELD_FLAGS` constant was deleted. The flag mask constants (`FLAG_MASK_COMMAND = 1 << 20`, `FLAG_MASK_ALTERNATE = 1 << 19`) were already numerically correct (`kCGEventFlagMaskCommand`/`kCGEventFlagMaskAlternate`) — only the source of the flags value was wrong.
- **Bug 3 (MEDIUM, fixed, high real-world impact): AX/Quartz coordinates are in points, but were being fed directly into Tauri's `PhysicalPosition` (device pixels) — this visibly mispositioned the bubble on essentially every real Mac, since Retina (scale factor 2) is the overwhelming majority.** The chain: `probe_selection` → `CGRect` in points → `AnchorPayload{x,y}` → `show_bubble`/`show_bubble_menu` → `w.set_position(PhysicalPosition::new(x, y))`. On a 2x Retina display this places the bubble at roughly half the intended on-screen position. A related inconsistency existed in `clamp_to_monitor`: the incoming `(x, y)`/`CGDisplayBounds` are points, but `window.outer_size()` (used for the clamp's `w`/`h`) returns physical pixels, over-subtracting near the right/bottom edge. A further related instance (found while fixing this, not explicitly named by the reviewer but the same root cause, in code added by this session's own earlier Gap 3 fix) was in `maybe_close_bubble_menu_on_outside_click`'s outside-click hit test: the incoming click `(x, y)` is points on macOS, but `w.outer_position()`/`w.outer_size()` are physical, so the "is this click inside the menu" bounds check was off by the same ~2x on Retina — a genuine outside click could still numerically read as "inside," so the bubble menu could fail to close. Fixed by converting consistently to logical (points) using each window's own `scale_factor()` before comparing/positioning, macOS-only (Windows already has matching units — UIA/`GetWindowRect` are both physical, so its paths are untouched): `clamp_to_monitor`'s macOS branch now converts `outer_size()` to logical via `.to_logical(scale)` before calling the shared `clamp_rect_to_monitor`; `show_bubble`/`show_bubble_menu` now call `w.set_position(LogicalPosition::new(x, y))` on macOS (still `PhysicalPosition` on other platforms); `maybe_close_bubble_menu_on_outside_click` now converts `outer_position()`/`outer_size()` to logical on macOS before the bounds comparison. `LogicalPosition`/`LogicalSize` were chosen over manual scale-factor multiplication because Cocoa's "points" are exactly what Tauri calls "logical" pixels on macOS — no arithmetic needed for the position values themselves, only for the borrowed `outer_size()`/`outer_position()` values that come back in physical pixels from Tauri regardless of platform.
- Verification: `cargo check`, `cargo build --lib --bins`, `cargo test --lib --bins` (19 tests, still all passing — none of these fixes touched the pure-logic functions the existing tests cover), and `cargo fmt -- --check` all pass cleanly from `src-tauri/`, including the final binary link (confirms the new `CGEventGetFlags` FFI declaration resolves). `npx tsc --noEmit` passes (no frontend changes).
- **Not verified — no real device/UI interaction from this environment.** All three fixes were made by static/logical reasoning (an independent critical review plus this session's own follow-up reasoning for the related outside-click instance), not by running the code against a real app. The Retina/coordinate-scaling fix (Bug 3) specifically needs real-device confirmation on an actual Retina Mac — nothing in any session so far has visually confirmed the bubble/menu land at the correct screen position, or that clicking genuinely outside the bubble menu now reliably closes it. See Known Gaps below.

### 2026-07-11 (macOS selection bubble — Phase 4 gap-closing pass)

Found the macOS selection watcher (previous "Phase 4 first pass" entry below) already in the working tree — implemented by an earlier pass this same session/day — and audited it directly against `roadmap-mac.md`'s Phase 4 spec and the Windows reference implementation it's meant to functionally match. It compiled/built/tested cleanly, but three real gaps existed between what was built and what the reference (`selection_watcher.rs`'s `mod win`) actually does. All three are now fixed, plus two research questions the spec asked to verify rather than assume (coordinate space, AX thread-affinity) were actually researched via web search rather than asserted from memory.

- **Gap 1 (fixed): no Electron/web-app AX fallback.** `probe_selection` (mac) only ever tried `kAXFocusedUIElement`; if that had no usable selection it returned `None` immediately, unlike the Windows backend's `probe_selection`, which falls back to `ElementFromPoint` at the last mouse-up point specifically for apps (Discord/WhatsApp/Slack/Notion-style Electron UIs) that leave focus on a wrapper element. Fixed: `selection_watcher.rs`'s mac `probe_selection` now tries the focused element first, then falls back to `AXUIElementCopyElementAtPosition(systemWideElement, x, y, &mut element)` using a newly-tracked last-mouse-up point (`LAST_MOUSE_UP_X`/`LAST_MOUSE_UP_Y`, captured in `tap_callback` via a new `CGEventGetLocation` call on `kCGEventLeftMouseUp`), then retries the same selection read on that element — mirroring the Windows fallback's structure and its `point.x != 0 || point.y != 0` "haven't seen a real click yet" guard.
- **Gap 2 (fixed): macOS multi-monitor clamping was a stub.** `lib.rs`'s `clamp_rect_to_monitor` macOS branch previously just returned the raw unclamped point with a `TODO(mac)` comment — meaning a bubble/menu positioned near a display edge (very plausible: the anchor is the bottom-right corner of a selection, which is often near the right edge of a narrow app window) could render partially or fully off-screen, and multi-monitor setups weren't handled at all. Fixed with a new `lib.rs::mod mac_display` using raw `CoreGraphics` FFI (`CGGetDisplaysWithPoint` to find which display contains the anchor point, `CGDisplayBounds` for that display's bounds) plus a pure `clamp_point_to_bounds` helper doing the exact same `x.clamp(min_x, max_x)` math the Windows branch uses. Note: this clamps against the display's *full* bounds, not a menu-bar/dock-excluded "work area" the way Windows' `GetMonitorInfoW` `rcWork` does — `CoreGraphics` has no direct equivalent, and the AppKit one (`NSScreen.visibleFrame`) is a main-thread-only call not worth an extra marshal for a small floating bubble; documented as a known minor limitation (a bubble very close to the menu bar/dock could still overlap it slightly).

- **Research finding — coordinate space (asked-for, not assumed):** confirmed via Apple's own published reference docs that `AXUIElementCopyElementAtPosition` takes "top-left relative screen coordinates" (primary display's top-left corner is (0,0), +x right, +y down) — the same Quartz/CoreGraphics global-display space `CGEventGetLocation` (mouse click locations) and `CGDisplayBounds` (used by the new `mac_display` module) use. `kAXBoundsForRangeParameterizedAttribute`'s returned `CGRect` is a screen rect in this same space. ### 2026-07-11 (macOS Escape-key dismissal — `esc_hook.rs` Phase 1)

Implemented the macOS equivalent of the Windows-only `esc_hook.rs` module (Phase 1 of `roadmap-mac.md`, the "global escape or dismissal handling" bullet). This was previously a real macOS UX gap: `Overlay.tsx`'s own `window.addEventListener("keydown", ...)` Escape handler only works if the overlay webview genuinely has OS keyboard focus, which isn't guaranteed (`show_overlay`'s `w.set_focus()` can fail silently or lose focus moments later), and unlike Windows there was no native fallback at all on macOS — a user could get stuck unable to dismiss the borderless overlay via Escape.

- `src-tauri/src/esc_hook.rs`: added a macOS backend using a `CGEventTap` (Quartz Event Services), the closest structural equivalent to the Windows module's `WH_KEYBOARD_LL` low-level keyboard hook — it observes (and, in `kCGEventTapOptionDefault`, can consume) keyboard events system-wide regardless of which app/window is focused, unlike `NSEvent` local/global monitors (local only sees this app's own windows; global can observe but not consume another app's keystrokes). Implemented via raw FFI directly against the `CoreFoundation` and `CoreGraphics` frameworks (same no-new-crate-dependency style `clipboard.rs` already uses for `AXIsProcessTrusted`/`AXIsProcessTrustedWithOptions`); `CoreGraphics` is already linked into the binary transitively via `enigo`'s macOS backend, so this added zero new Cargo.toml dependencies.
  - The entire original Windows implementation was moved verbatim into a `#[cfg(target_os = "windows")] mod win { ... }` (re-exported via `pub use win::{start, stop};`) with no internal changes — purely a wrapping move so the file can compile for both targets; behavior is unchanged.
  - The new `#[cfg(target_os = "macos")] mod mac { ... }` (also re-exported as `pub use mac::{start, stop};`) exposes the same `pub fn start(app: &AppHandle)` / `pub fn stop()` API. `start` is idempotent via a `Mutex<TapState>` state machine (`Idle` / `Starting` / `Running(run_loop_ptr)`) rather than a bare atomic — chosen specifically to avoid repeating the exact hook-thread-restart race described in the Windows module's own doc comment (two threads touching one shared handle could silently unhook a freshly-installed hook). The mutex-guarded state transitions make a new tap thread provably unable to spawn until the previous one has fully torn down (`CGEventTapEnable(false)` + `CFMachPortInvalidate` + `CFRelease`s completed), and a thread whose `start()` was immediately followed by `stop()` mid-startup tears itself down without ever entering `CFRunLoopRun()`.
  - Accessibility-permission handling: `start()` checks `clipboard::accessibility_trusted(false)` (the existing non-prompting check, now called from a second call site) on every call and simply skips installing the tap if permission isn't granted — no error, no crash, just a trace log — leaving `Overlay.tsx`'s frontend listener as the only Escape path, exactly as it behaved before this module existed. Because `start()` runs on every overlay show (same call site as Windows), permission granted later in the same session (e.g. via the Phase 2 `AccessibilityView.tsx` tutorial) is picked up automatically on the very next overlay show with no additional polling.
  - Thread-safety / main-thread discipline: the tap's `CFRunLoop` runs on a dedicated thread spawned by `run_tap_thread` (mirroring the Windows module's `std::thread::spawn(run_hook_thread)` pattern), never the Tauri/AppKit main thread. The tap callback (`tap_callback`) is kept deliberately trivial per Apple's guidance (a slow callback risks the OS silently disabling the tap via `kCGEventTapDisabledByTimeout`): it only reads the keycode via `CGEventGetIntegerValueField` and decides whether to consume the event. All Tauri/webview/AppKit work — checking `w.is_visible()`, calling `stop()`, calling `app.emit_to("overlay", "overlay:esc", ())` — is marshaled onto the main thread via `app.run_on_main_thread(...)`, fire-and-forget, following the exact same pattern established in `foreground.rs`/`lib.rs`'s prior macOS main-thread crash fixes (see the two "macOS foreground-app/paste-target" Recent Updates entries below). The callback consumes Escape unconditionally whenever the tap is installed (trusting that `start()`/`stop()`'s lifecycle — scoped to `show_overlay` / the overlay's `CloseRequested` handler — already means "the overlay is meant to be visible right now"), matching the Windows hook's `return 1` swallow behavior, rather than doing a second synchronous main-thread round trip inside the callback to double check visibility first (which would risk the same timeout-disables-the-tap failure mode the task explicitly warned about).
  - `kCGEventTapDisabledByTimeout` / `kCGEventTapDisabledByUserInput` are handled by re-enabling the tap, so a transient OS-triggered disable doesn't permanently kill Escape for the rest of the process's life. (The first version of this re-enable call had a bug — see the "critical-review fixes" entry immediately below.)
- `src-tauri/src/lib.rs`: `#[cfg(target_os = "windows")] pub mod esc_hook;` → `#[cfg(any(target_os = "windows", target_os = "macos"))] pub mod esc_hook;`. The two call sites (`show_overlay`'s `esc_hook::start(&handle)` and the pre-warmed overlay window's `CloseRequested` handler's `esc_hook::stop()`) both changed from `#[cfg(target_os = "windows")]` to `#[cfg(any(target_os = "windows", target_os = "macos"))]`. No other lib.rs changes; Windows behavior is unchanged.
- Verification: `cargo check`, `cargo build --lib --bins`, and `cargo test --lib --bins` (from `src-tauri/`, on this macOS dev machine, compiling the real `target_os = "macos"` branch) all pass cleanly — including the final binary link step, which confirms the raw `#[link(name = "CoreFoundation"/"CoreGraphics", kind = "framework")]` FFI declarations resolved correctly against the real system frameworks (a link failure would have surfaced there, not at `cargo check`).

Nothing here confirms: that `CGEventTapCreate` actually succeeds at runtime (vs. just type-checking) when Accessibility is genuinely granted; that pressing Escape while the overlay is shown and NOT focused (e.g. right after a `set_focus()` that silently failed, or with another app frontmost) actually dismisses it; that the tap correctly stays silent/inert (no accidental Escape-swallowing in other apps) when the overlay isn't visible; that the graceful-degradation path (Accessibility not granted → tap skipped, frontend listener still works when focused) behaves as designed; or that a revoke-while-running scenario doesn't leave a stale tap installed. ### 2026-07-11 (macOS Escape-key dismissal — critical-review fixes)

A critical review of the `CGEventTap` implementation above (previous entry) found two real, high-severity FFI-lifecycle bugs and one low-severity robustness note. All fixed in `src-tauri/src/esc_hook.rs`'s `mod mac`; no other files changed.

- **Bug 1 (HIGH, fixed): the disabled-by-timeout/user-input re-enable path called `CGEventTapEnable(proxy as CFMachPortRef, true)`, but `proxy` (the callback's `CGEventTapProxy` parameter) is a distinct, opaque token — Apple documents it as valid only for `CGEventTapPostEvent`, not as a substitute for the actual `CFMachPortRef` `CGEventTapCreate` returned. The `as CFMachPortRef` cast compiled and silenced the type system without making the value correct.** Concretely: macOS auto-disables a tap after a slow callback or certain input bursts — a realistic trigger, since the very first `run_on_main_thread` marshal in the callback could be slow if the main loop/webview is busy right at overlay-show time. Once disabled, the broken re-enable call would fail silently, and because `start()`'s `TapState` still said `Running` (it had no way to learn the OS-level tap had gone dark), it would never reinstall on a later overlay show — native Escape dismissal would be dead for the rest of the process's lifetime after a single transient disable, with no user-visible error. Fixed by adding a new static, `TAP_REF_SLOT: AtomicPtr<c_void>`, and threading a pointer to it through `CGEventTapCreate`'s `user_info` parameter (previously `std::ptr::null_mut()`, unused). Since the real tap handle isn't known until `CGEventTapCreate` returns — it can't be baked into `user_info` at call time — `run_tap_thread` fills `TAP_REF_SLOT` with the real handle immediately after creation succeeds (before the tap is enabled, so the callback can't observe it empty), and `tap_callback` now reads the real handle back out of `user_info` for the re-enable call instead of touching `proxy` at all (`proxy` is now `_proxy`, unused).
- **Bug 2 (HIGH, fixed): a lost-`CFRunLoopStop` race could leave a zombie `CGEventTap` swallowing Escape system-wide (every app on the machine, not just reWrite) and leak a second tap thread on top of it.** `run_tap_thread` published `TapState::Running(rl)` before calling the old blocking `CFRunLoopRun()`; `CFRunLoopStop` is documented to only take effect while the run loop has an active run frame, so a `stop()` call landing in the narrow window between "state says Running" and "`CFRunLoopRun()` has actually started spinning" was silently a no-op. Trigger: a fast overlay open-then-close (hotkey immediately followed by Escape or a click-away) could hit exactly this window — the thread would then block in `CFRunLoopRun()` forever with the tap still enabled, while `stop()` (unaware its stop had been dropped) had already reset shared state to `Idle`, so the *next* overlay show's `start()` would spawn an unrelated second tap thread on top of the still-live first one. Fixed by replacing the single blocking `CFRunLoopRun()` with a bounded poll loop: `CFRunLoopRunInMode(kCFRunLoopDefaultMode, 0.25, false)` in a `while`-style loop that re-checks a new `STOP_REQUESTED: AtomicBool` (and, defensively, the `TapState` itself) after every 250ms slice, breaking into teardown as soon as either says stop. `stop()` still calls `CFRunLoopStop` too (a harmless, often-successful fast path when timing allows), but termination no longer *depends* on it — worst case, a lost `CFRunLoopStop` now just costs up to ~250ms instead of hanging forever. `STOP_REQUESTED`'s writes (by `stop()`, before flipping state to `Idle`) and its reset-to-`false` (by `run_tap_thread`, right after publishing `Running`) both happen inside the same `TAP` mutex critical section as the corresponding `TapState` write, specifically so the flag and the state can never be observed out of sync with each other — ruling out a variant race where a reset-to-`false` could stomp a stop request that arrived a moment earlier.
- **Low-severity note (fixed, one line): teardown order relied implicitly on `CFMachPortInvalidate` auto-detaching the derived run-loop source rather than removing it explicitly first.** Added an explicit `CFRunLoopRemoveSource(rl, source, kCFRunLoopCommonModes)` call before `CFMachPortInvalidate`/the `CFRelease`s, so correct teardown ordering doesn't depend on that implicit (if correct) behavior surviving future edits.
- Confirmed by the reviewer and left unchanged: the callback ABI signature (matches `CGEventTapCallBack` exactly), all FFI constants (verified against the CoreGraphics/CGEventTypes headers), the main-thread marshaling discipline (the callback still never touches Tauri/AppKit directly, only via `run_on_main_thread`), the Accessibility-not-granted degradation path, the `Mutex<TapState>` preventing concurrent double-`start()`, and the Windows `mod win { ... }` wrapping move (still verbatim, still behavior-preserving).
- Verification: `cargo check`, `cargo build --lib --bins`, `cargo test --lib --bins` (from `src-tauri/`) all pass cleanly after these fixes, including the final binary link (confirms the new `CFRunLoopRunInMode`/`CFRunLoopRemoveSource`/`kCFRunLoopDefaultMode` FFI declarations resolve against the real frameworks).
- **Still not verified — no real device/UI interaction from this environment.** These were found and fixed by static/logical reasoning about the FFI and concurrency, not by running the code, so real-device verification is still required and the ask is unchanged from the entry above: press Escape against an unfocused overlay, confirm graceful degradation without Accessibility, and confirm a live grant/revoke cycle. Two additional things worth specifically exercising now given what was fixed here: (a) rapid hotkey-then-Escape / hotkey-then-click-away cycles in a tight loop, watching the trace log for `CGEventTapCreate failed` (would indicate a leaked/uninvalidated prior tap) and confirming Escape still dismisses a *later* overlay show normally (would indicate a zombie tap from Bug 2 recurring); (b) if a real disabled-by-timeout notification can be provoked (e.g. by artificially stalling the main thread during the `run_on_main_thread` marshal), confirming the trace log's `tap disabled by OS, re-enabling` line is followed by Escape continuing to work afterward, not going permanently dead (Bug 1 recurring).

A critical review of the Accessibility permission tutorial (previous entry below) found five issues; four required code changes, all now fixed. All fixes are frontend-only (`src/pages/AccessibilityView.tsx`, `src/pages/Settings.tsx`); no backend/Rust changes were needed for this pass.

Neither Home nor the Accessibility view (nor the sidebar, which reads `accessibilityGranted`) renders until both checks have resolved, so the first real paint is already correct.

## Product Summary

reWrite is a system-wide AI text transformer. The user highlights text in another app, invokes reWrite through a hotkey or passive selection bubble, chooses a rewrite skill, and the rewritten result replaces the original selection.

Main user flows:

- Overlay hotkey: capture selected text, show full skill picker, rewrite, paste.
- Super hotkey: capture selected text, apply default skill silently, paste.
- Selection bubble: detect highlighted text, show a small bubble, open compact skill menu, rewrite, paste.
- Settings: manage auth, billing, usage, hotkeys, default skill, custom skills, history, and preferences.

## Main Data Flow

### Overlay Hotkey

1. Global shortcut fires in `lib.rs::on_hotkey`.
2. App remembers the paste target window and output format.
3. `clipboard::capture_selection` simulates copy and stores selected text/original clipboard in `AppState`.
4. `show_overlay` displays `Overlay.tsx`.
5. `Overlay.tsx` calls `rewrite_with_skill`.
6. `rewrite_with_skill` builds prompt, calls Supabase rewrite, logs history, updates usage.
7. `Overlay.tsx` calls `paste_text`.
8. `paste_text` hides overlay, refocuses source window, calls clipboard paste/restore.

### Super Hotkey

1. Global shortcut fires in `lib.rs::on_super_hotkey`.
2. App captures selection.
3. App applies default skill without showing overlay.
4. App shows processing UI while the request runs.
5. App pastes result and logs history.

### Selection Bubble

1. `selection_watcher` detects mouse/keyboard events that may affect text selection.
2. UI Automation confirms selected text and selection rectangle.
3. Rust emits selection events and calls `show_bubble`.
4. `Bubble.tsx` calls `bubble_clicked`.
5. `bubble_clicked` stores selected text and opens `BubbleMenu.tsx`.
6. `BubbleMenu.tsx` calls `rewrite_with_skill`, then `paste_text`.

### Auth and Billing

1. Magic link starts through `send_magic_link`.
2. Deep link is parsed by `handle_deep_link` and saved with `auth::save_session`.
3. `ensure_valid_token` refreshes tokens when needed.
4. `sync_subscription` caches plan/subscription state.
5. Rewrite calls are authenticated and usage-limited by Supabase.
6. Checkout and portal URLs are created by Supabase functions and opened externally.

- Mac support is no longer purely planning-stage: Accessibility onboarding, hotkey capture/paste plumbing, Escape dismissal, and a first pass at passive selection detection are implemented. The macOS flows are still **not release-verified** on a real packaged app/device.
- **macOS selection bubble (`selection_watcher.rs` mac backend) is code-complete, including the 2026-07-11 gap-closing pass AND a same-day critical-review pass that fixed three more real bugs, but STILL UNVERIFIED ON A REAL DEVICE.** It compiles, links, and passes tests/builds (19 unit tests), but no real app has been driven interactively from this environment. A 2026-07-11 follow-up fixed the first-run/recovery bug where Accessibility could be granted after startup while the watcher remained stopped. A second pass added the Electron/web-app AX fallback, real `CGDisplayBounds`-based multi-monitor clamping (`lib.rs::mod mac_display`), and click-outside-closes-menu parity with Windows. A third pass (independent critical review) then found and fixed: (1) a HIGH-severity start/stop race — bare atomics were replaced with the same `Mutex<TapState>` generation-tracking state machine already proven in `esc_hook.rs`'s mac backend, closing a lost-stop-revival / stale-generation-stomp race that a fast bubble-toggle or Accessibility revoke→grant cycle could trigger (two live taps, or the bubble going silently dead for the session); (2) Cmd+A (select-all) detection was reading the wrong `CGEventField` (no such field exists for modifier flags) and so never actually worked — fixed by switching to the correct `CGEventGetFlags` accessor; (3) AX/Quartz coordinates (points) were being fed directly into Tauri's `PhysicalPosition` (device pixels), which would visibly misposition the bubble/menu on every Retina Mac (~half the intended position) — fixed by using `LogicalPosition`/`.to_logical()` conversions consistently at every Tauri window-position/size touchpoint on macOS (`show_bubble`, `show_bubble_menu`, `clamp_to_monitor`, and `maybe_close_bubble_menu_on_outside_click`'s hit-test, the last being a related instance of the same bug found in this session's own earlier code while fixing the reviewer's report). A human still needs to: grant Accessibility, highlight text in Notes/Mail/Safari/Chrome/Slack/Notion/Word/VS Code, confirm `selection:detected` appears in logs, confirm the visible bubble lands at the CORRECT position near the selected text on a real Retina display (this is the single most important unverified claim now — Bug 3 was a real, high-impact positioning bug, and its fix is unverified), confirm it stays fully on-screen near a display edge / on a secondary monitor, click it, choose a skill, confirm paste-back replaces the original selection, click outside an open bubble menu and confirm it now closes, drag-select new text while a bubble menu is open to confirm the drag-vs-click distinction behaves as intended, select-all with Cmd+A from a fresh (no prior drag) focus to confirm the bubble now appears, and rapidly toggle the "Selection bubble" Settings switch off/on several times in a row watching the trace log for `CGEventTapCreate failed` or any sign of two simultaneous taps (regression check for Bug 1). None of them use AppKit's `NSScreen`/`NSView` bottom-left-origin "flipped" space. **Refinement from the critical-review pass:** that shared space is in *points*, not raw device pixels — which is exactly what Tauri calls "logical" (not "physical") position/size on macOS. The origin/flip-direction question above was never wrong, but the points-vs-pixels question was originally missed and caused the real Bug 3 above; conversion IS needed at the boundary into Tauri's `PhysicalPosition`/`PhysicalSize` APIs, via each window's `scale_factor()`. See the sourcing/reasoning documented inline in `selection_watcher.rs`'s `mod mac` and `lib.rs`'s `mod mac_display`/`clamp_to_monitor`/`show_bubble`/`maybe_close_bubble_menu_on_outside_click`.
  - **AX thread-affinity finding (researched, but not 100% certain either way):** no official Apple documentation was found stating explicitly that `AXUIElementCopyAttributeValue`/`AXUIElementCopyElementAtPosition` are safe to call off the main thread — but none was found requiring the main thread either (unlike `AXObserver`, whose notification delivery explicitly requires an active run loop). Hammerspoon's `libaxuielement.m` — a mature, widely-used AX-heavy tool — calls these same functions directly with no main-thread marshaling anywhere in the file, and architecturally they're synchronous Mach-IPC calls to another process's accessibility server rather than calls into this process's own AppKit/window-server state (the class of call that does need `run_on_main_thread` elsewhere in this codebase). `probe_selection` is called unmarshaled from the worker thread on this basis. If a real device later shows AX calls behaving strangely off the main thread (hangs, garbage data, crashes) specifically when called from `selection_watcher`'s worker thread, revisit this assumption first.
- **macOS foreground-app/paste-target AppKit calls are marshaled onto the main thread, but the crash fix is UNVERIFIED AT RUNTIME — read this before assuming it's resolved.** `foreground::detect`, `remember_paste_target_window`, and `focus_paste_target_window` all used to call AppKit (`NSWorkspace`/`NSRunningApplication`) directly from `on_hotkey`/`on_super_hotkey` or the async tasks/commands they spawn, which are not guaranteed to run on the main thread; that was blamed for a crash on the super hotkey, and `detect()` was stubbed to always return `PlainText` as a stopgap. All three now marshal the AppKit call through `app.run_on_main_thread` (channel-and-block for `detect`, since its result is needed synchronously; fire-and-forget for the other two, since their results are only read later). This compiles and passes unit tests, and the channel-based hop in `detect` is provably deadlock-free (traced `tauri-runtime-wry`'s `send_user_message`: it runs the posted closure inline, on the calling thread, when already on the main thread — so the channel send always happens before the `recv_timeout` wait, whichever thread `detect` is called from).

Tracing the vendored `global-hotkey` 0.8.0 macOS backend (`InstallEventHandler(GetApplicationEventTarget(), ...)` + `GlobalHotKeyEvent::send`, which `tauri-plugin-global-shortcut` wires straight to its `.with_handler()` callback via `set_event_handler`) suggests the hotkey handler callback — i.e. `on_hotkey`/`on_super_hotkey` themselves — already runs inline on the main thread on macOS. If that's right, `run_on_main_thread` is a no-op here (it detects "already on main thread" and just calls the closure directly), the AppKit call was always happening on the main thread, and the original crash's real cause is still unknown (candidates: `objc2`/`objc2-app-kit` 0.2.x version quirks, missing/incorrect autorelease pool handling around `Retained<T>` across the Carbon callback boundary, or something else entirely). This was not tested by actually running the packaged app and firing the hotkey from this session — that would send real synthetic keystrokes and steal focus on what may be a live machine, which needs a human driving it interactively, not an automated session.
  - **How to actually verify, next time a human runs the app:** every `trace()` line already logs `tid=<ThreadId>`. Two new diagnostic lines were added for this specifically: `on_hotkey: enter is_main=<bool>` / `on_super_hotkey: enter is_main=<bool>` at the very top of each handler in `lib.rs` (before anything else runs), and inside `foreground::detect_impl`'s macOS branch, `foreground::detect_impl: calling thread is_main=<bool>` (logged before `run_on_main_thread` is called) followed by `foreground::detect_impl: run_on_main_thread closure is_main=<bool>` (logged from inside the closure, i.e. wherever it actually executed). `is_main` is computed by `lib.rs::is_main_thread()` against a `ThreadId` captured once at the very start of `run()`. Run the app, press the super hotkey a few times against Notes or Mail, and read the trace log (stderr):
    - If `on_hotkey`/`on_super_hotkey`'s `is_main` is already `true` and the two `detect_impl` lines show the same thread id, the marshaling was a no-op — no crash without it would mean the original bug had a different, still-unfixed cause; a crash even now would mean this fix doesn't address it either, and needs new investigation (autorelease pools, `objc2` version bump, etc.).
    - If `on_hotkey`/`on_super_hotkey`'s `is_main` is `false` and the `detect_impl` closure's thread id differs from the caller's, the marshaling is doing real work and this fix is plausibly the actual crash fix — confirm by pressing the hotkey repeatedly with no crash, and check pasted output into Notes/Mail comes through as HTML (bold/rich), not plain text.
