## What this feature is

Today reWrite is 100% hotkey-driven: press a hotkey, it simulates Ctrl+C, shows the full skill-picker overlay. v1.1.0 adds a passive, Grammarly-style flow on top of that (unchanged) flow:

1. Highlight text in any app.
2. A tiny (10x10px) bubble appears at the end of the selection, with no hotkey needed.
3. Click it → a small titles-only dropdown of skills appears next to it.
4. Pick a skill → runs through the exact same rewrite pipeline the hotkey path already uses.

## Key design decisions made this session

- **Trigger cheaply, confirm precisely.** A low-level `WH_MOUSE_LL` mouse hook (same idempotent start/stop pattern as the existing `esc_hook.rs`) detects drag-release as a cheap trigger. Windows UI Automation (`TextPattern`) then confirms a real non-empty text selection and gives its exact bounding rect. This is what keeps the bubble from popping up during RTS unit-boxing or Explorer/Photoshop drag-drop — games generally don't implement UIA, and file/canvas drag isn't a `TextPattern` selection, so no denylist is needed.
- **UIA supplies the text too, not just detection.** Originally the plan was to re-run the existing clipboard Ctrl+C-simulate capture when the bubble is clicked. That was changed mid-build: clicking the bubble window likely shifts OS foreground focus away from the source app, which would make a fresh Ctrl+C simulate copy the wrong thing. Since UIA's `probe_selection()` already extracts the selected text as part of detection, the bubble's click handler uses that text directly (via the `selection:detected` event payload) and only does a **passive** clipboard read (`clipboard::snapshot_clipboard()`, no synthetic copy) to snapshot the pre-existing clipboard contents for the optional restore-after-paste behavior.
- **`windows` crate, not `windows-sys`, for UIA.** `windows-sys`'s `Win32_UI_Accessibility` feature exposes every UIA COM interface as a bare `*mut c_void` with no generated vtables/methods — unusable without hand-rolling vtable structs. The `windows` crate generates real method bodies. It was already present transitively at 0.61.x, so this added no new major dependency version.
- **Bubble ships on by default** (`config.bubble_enabled`, default `true`), with a Settings toggle to disable it — this was an explicit user decision this session, made knowingly aware of the "what if it fires during games" risk (mitigated by the UIA-based detection above).

### Sprint 1 — Passive selection watcher service
- New `src-tauri/src/selection_watcher.rs`: idempotent `start()`/`stop()` (mirrors `esc_hook.rs`), a `WH_MOUSE_LL` hook thread whose callback does the absolute minimum (never calls UIA on the hook thread — critical for not adding input lag in games), and a separate worker thread that debounces ~200ms then runs the UIA probe.

## Post-handoff fixes (cross-app manual test findings)

Two bugs reported from real cross-app testing:

1. **Invisible in VS Code.** The original dot was a near-black 10x10 circle (`#16161a`) — indistinguishable against a dark editor theme. Fixed in `src/pages/Bubble.tsx`: now a white circle with a slow-rotating conic-gradient outline (blue → green → yellow → red, `rw-spin` keyframe from `src/index.css`, 6s linear loop), bumped to 16x16 (`src-tauri/src/lib.rs`'s bubble window `inner_size`) to fit the ring. **Caveat not yet verified:** VS Code's Monaco editor may not expose a real UIA `TextPattern` selection at all unless `editor.accessibilitySupport` is `"on"` (it defaults to `"auto"`, which only turns on the full accessibility tree when a screen reader is detected). If the bubble still never appears in VS Code after this color fix, that setting is the next thing to check — it'd mean detection itself is the problem, not contrast.
2. **Tiny/unresponsive in WhatsApp.** Root cause unconfirmed without live repro, but two real hardening gaps were found and fixed in `show_bubble`/`show_bubble_menu` (`src-tauri/src/lib.rs`): neither was force-reasserting topmost z-order before showing, so an app that's itself always-on-top (WhatsApp's floating call window, some overlay-style apps) could end up rendered above our bubble/menu — visually absent or unclickable even though the command fired correctly. Both now toggle `set_always_on_top(false)` → `set_always_on_top(true)` right before `.show()` to force to the top of the topmost band. The enlarged 16x16 hit target (see #1) should also help, since the original 10x10 target was easy to miss-click near dense text.

**Still not fully diagnosed:** if WhatsApp clicks still do nothing after these fixes, next step is adding temporary `crate::trace(...)` calls in `bubble_clicked` and `show_bubble_menu` to confirm the command actually fires and see the coordinates it's using — would need a human to reproduce with the dev console/log open.

## Second round of fixes — found the real menu bug

After the color/size fix, the bubble itself was confirmed visible and clickable, which narrowed things down. Two more issues reported and root-caused in `src-tauri/src/selection_watcher.rs`:

1. **Bubble didn't disappear when the selected text was deleted.** Root cause: the mouse hook is the *only* thing that ever re-triggers a probe cycle. Deleting a selection via keyboard (Delete/Backspace/typing over it) never fires a mouse event, so `selection:cleared` was never emitted and the bubble just sat there pointing at text that no longer existed. Fixed by adding a second low-level hook, `WH_KEYBOARD_LL` (`keyboard_hook_proc`), installed on the same hook thread alongside the existing `WH_MOUSE_LL` one. It's gated on a new `HAD_SELECTION` atomic (mirroring the worker's local `had_selection`) so it's a no-op — doesn't even touch the channel — whenever there's no active selection to lose, keeping the "never add latency to games/typing" property the mouse hook already had. The `WorkerMsg::MouseUp` variant was renamed to `WorkerMsg::Recheck` since it's now shared by both hooks.

2. **The actual "menu never appears" bug.** This was a real race, not a red herring: clicking the bubble is itself a `WM_LBUTTONUP` the global mouse hook sees, same as any other click. ~200ms later (the existing debounce), the worker's probe cycle ran again — and most apps (WhatsApp included) don't clear their internal text selection just because a different window took focus, so UIA still reported the *same* selection as present. That re-emitted `selection:detected`, which re-called `show_bubble(...)` — popping the tiny dot right back up on top of (or instead of) the skill menu that had just opened, milliseconds after it appeared. This got worse, not better, after round 1 added the `set_always_on_top` re-toggle trick to `show_bubble`, since that made the reappearing bubble reliably win the z-order fight against the menu. Fixed with a targeted guard: `run_probe_cycle` now checks `bubble_menu_is_open()` (literally `get_webview_window("bubble_menu").is_visible()`) first and skips the entire probe while the menu is open, regardless of what UIA would've reported. This is a single-source-of-truth check — it doesn't matter *how* the menu was shown/hidden, only whether it's currently visible.

Both fixes are in `selection_watcher.rs` only; no changes needed to `Bubble.tsx`/`BubbleMenu.tsx`/`commands.rs` for these two. `cargo build` and `npm run build` both clean after this round. Still not independently verified live — needs another manual pass (delete a selection via keyboard while the bubble is showing; click the bubble in WhatsApp and confirm the menu now stays open).

## Third round — menu opens then instantly closes itself

After round 2, reported behavior narrowed further: click the bubble, it disappears (so `bubble_clicked` is definitely firing), but the menu never visibly appears at all — not even a flash.

Root cause: `BubbleMenu.tsx`'s dismiss-on-blur listener. Its original comment claimed it "mirrors Overlay.tsx's pattern", but that's wrong — checked `Overlay.tsx` directly and it has **no** close-on-blur logic at all (it only reacts to `focused === true`, to refresh data; blur does nothing, closing is only ever explicit via the X button/Esc/hotkey). `BubbleMenu.tsx`'s blur handler, by contrast, called `close_bubble_menu` synchronously off the very first `focused: false` payload. `show_bubble_menu` (`lib.rs`) does `set_position` → `set_always_on_top` toggle → `.show()` → `.set_focus()` on a window that was hidden a moment before; that sequence can produce a transient/out-of-order activation blip on Windows (a `false` arriving right around the real `true`), which the old code would treat as "user clicked away" and close the menu immediately — before it was ever visibly rendered. This is exactly "bubble disappears, no menu" from a single click, with no cross-app interaction needed to trigger it.

Fixed in `BubbleMenu.tsx`: the blur handler no longer closes synchronously off the raw event. On a `focused: false` payload it now waits 150ms and then re-checks `getCurrentWindow().isFocused()` before actually calling `close_bubble_menu` — a transient blip self-corrects within that window and no longer causes a close; a real "user clicked back into the source app" still closes it, just ~150ms later.

Also added, as a safety net in case this isn't the whole story: `crate::trace(...)` calls in `bubble_clicked`, `show_bubble_menu`, and `close_bubble_menu` (`commands.rs`/`lib.rs`). These print to the terminal running `cargo run` (dev build only) — if the menu still misbehaves, capture that terminal output during a repro; it'll show whether `show_bubble_menu` actually ran, whether `.show()`/`.set_focus()` reported success, and whether/when something called `close_bubble_menu` afterward.

`cargo build` and `npm run build` both clean. Not yet independently verified live.
